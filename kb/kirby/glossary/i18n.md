# I18n / translations (aliases: `t()`, `tc()`, `tt()`, “language variables”)

## Meaning

In Kirby, “i18n” usually refers to translating **UI strings and custom text snippets** via language variables (not translating page content fields). Language variables live in language definition files and are accessed via helpers like `t()`, `tc()` (pluralization), and `tt()` (placeholders).

Plugins can also register translation strings, but core translations and plugin translations cannot be overridden from the Panel language editor.

## In prompts (what it usually implies)

- “Use `t('key')`” means: define `translations` in language files (or plugin translations) and then call `t()`.
- “Pluralize based on count” means: `tc('errors', $count)`.
- “Text with placeholders” means: `tt('alert', ['message' => '…'])` with `{ message }` placeholders.

## Variants / aliases

- Language variables stored in `site/languages/<code>.php` under `translations`
- Helpers:
  - `t('confirm', 'Confirm')` (fallback)
  - `tc('errors', 2)` (count)
  - `tt('alert', ['message' => '...'])` (placeholders)
- Plugins can register translation extensions (good for reusable blocks/plugins)

## Example

Language file snippet:

```php
return [
  'code' => 'en',
  'translations' => [
    'confirm' => 'Confirm',
    'errors'  => ['No errors', 'There is { count } error.', 'There are { count } errors.'],
  ],
];
```

Template usage:

```php
<?= t('confirm', 'Confirm') ?>
<?= tc('errors', 5) ?>
```

## MCP: Inspect/verify

- Use `kirby_roots` to locate the `languages` root, then inspect language files for `translations`.
- Use `kirby_eval` to verify runtime translation output:
  - example: `return [t('confirm', 'Confirm'), tc('errors', 2)];`
- If translations are plugin-provided, check the extension reference via `kirby://extension/translations` and list installed plugins with `kirby_plugins_index`.
